KALIANG Frontend — Полный Backend Dependency Audit
A. Executive Summary
Фронтенд представляет собой классический SPA-стек на ванильном JS без бандлера. Все HTTP-вызовы централизованы через единый файл api.js. Аутентификация реализована через двухтокенную схему: accessToken хранится in-memory (защита от XSS), refreshToken — в httpOnly-cookie. CSRF защита через выделенный endpoint. Реалтайм — через SSE (/events), один endpoint для обоих ролей (admin/user), разделение по именам событий. Портфолио полностью статично, к backend не обращается. Конфигуратор обращается к backend только при submit заказа через createTicket.

Критические вещи для backend-команды:

SSE endpoint /events должен работать с credentials: 'include' + httpOnly cookie — нестандартно для большинства SSE-реализаций.
CSRF endpoint /auth/csrf вызывается лениво перед каждым мутирующим запросом. Если он медленный — каждый POST будет иметь лишний round-trip.
Login и Register ожидают { accessToken, user } в теле ответа. Refresh ожидает только { accessToken }. Это разные shapes.
Endpoint /auth/me должен возвращать { user } с полем isAdmin (boolean) — от него зависит роутинг.
createTicket принимает опциональный orderConfig — полный объект конфигуратора. Backend должен его либо сохранять, либо игнорировать, но не падать.

C. Authentication & Session
Механика

1. Анонимный пользователь → auth.html
2. POST /auth/login → { accessToken, user }
   - accessToken → inMemoryAccessToken (JS variable, не персистируется)
   - user → sessionStorage['user'] (НЕ localStorage)
   - refreshToken → httpOnly cookie (устанавливается backend'ом в Set-Cookie)
3. Каждый следующий запрос:
   - Authorization: Bearer <inMemoryAccessToken>
   - credentials: 'include' (cookie автоматически)
4. На 401/403: POST /auth/refresh (только cookie, без тела)
   → новый accessToken → повтор оригинального запроса
5. Если refresh провалился → logout(): inMemoryAccessToken = null, csrfToken = null,
   sessionStorage.removeItem('user'), redirect /auth.html
6. Logout: POST /auth/logout → тот же logout()
Хранение
Что	Где	Lifetime
accessToken	JS-переменная (inMemoryAccessToken)	До перезагрузки страницы
refreshToken	httpOnly cookie (backend)	Backend-определяется
user (username, isAdmin, telegram_*)	sessionStorage['user']	До закрытия вкладки
csrfToken	JS-переменная	До перезагрузки
kaliang_lang	localStorage	Persistent
pendingOrder	sessionStorage['pendingOrder']	Temporary (cleared after restore)
Ролевая маршрутизация

checkAuth() → user.isAdmin === true  → admin/dashboard.html
checkAuth() → user.isAdmin === false → tickets.html
checkAuth() → null                   → /auth.html
tickets.js: если user.isAdmin → redirect на admin/dashboard.html
admin/dashboard.js: если !user.isAdmin → redirect на ../tickets.html
CSRF
GET /auth/csrf — вызывается лениво перед каждым не-GET запросом
Токен кешируется in-memory в csrfToken
Передаётся как заголовок: X-CSRF-Token: <token>
При logout/refresh — csrfToken = null (сбрасывается)
Redirect с return URL
При незалогиненном submit в конфигураторе:

js

sessionStorage.setItem('pendingOrder', JSON.stringify(config));
window.location.href = '/auth.html?returnTo=configurator.html';
Параметр returnTo в URL — frontend его не читает и не использует для автоматического редиректа. Это dead feature — pending order восстанавливается через sessionStorage при загрузке конфигуратора.

Особенности SSE
sse.onerror = () => {} — ошибки игнорируются, браузер переподключается автоматически
Нет явного close на logout — SSE соединение не закрывается при вызове logout(). Это потенциальная утечка
initSSE() вызывается только если inMemoryAccessToken есть в момент вызова
SSE не имеет heartbeat/ping механики на стороне фронтенда
Auth: полностью через cookie (withCredentials: true). Если cookie истечёт во время SSE-соединения — следующий reconnect после onerror получит 401, но frontend этого не обработает (onerror пустой)
CORS: endpoint /events должен разрешать credentials: true с origin https://telegram-bots.pl

H. Backend Questions / Open Issues
Обязательные ответы
Cookie flags: какие именно флаги на refreshToken cookie?

HttpOnly: true — необходим (FE не должен читать)
Secure: true — необходим в production
SameSite: ? — если Strict, SSE с cross-origin может не отправить cookie; если None — требует Secure; если Lax — POST cross-origin не отправит cookie
Recommendation: SameSite=Lax для refresh + SameSite=None; Secure для SSE если домены разные
CORS policy: CSP в HTML явно разрешает connect-src https://telegram-bots-backend.onrender.com. Нужно:

Access-Control-Allow-Origin: https://telegram-bots.pl (не *)
Access-Control-Allow-Credentials: true
Access-Control-Allow-Headers: Authorization, Content-Type, X-CSRF-Token
Access-Control-Allow-Methods: GET, POST, PATCH, OPTIONS
SSE auth: endpoint /events должен авторизовать через cookie, так как EventSource не поддерживает кастомные заголовки. Backend должен валидировать refreshToken или accessToken из cookie — уточнить что именно.

/auth/me response shape: должно включать telegram_chat_id и telegram_notifications_enabled напрямую в user объекте (не отдельный endpoint). Подтвердить.

/auth/csrf caching: backend должен принимать один и тот же CSRF токен на протяжении всей сессии (до logout). Нельзя ротировать токен по каждому запросу.

204 vs 200: все endpoints должны возвращать JSON (хотя бы {}). FE всегда вызывает response.json().

assignedAdminId в PATCH /admin/tickets/:id: FE отправляет camelCase (assignedAdminId). Backend должен принять именно это имя, или нужно выровнять.

Статус-коды: FE обрабатывает специально: 400, 401, 403, 404, 409, 423, 429, 500, 502, 503. Для корректного отображения сообщений — backend должен использовать именно эти коды.

Error shape: FE ожидает { error: string } в теле ошибки. Подтвердить что все ошибки имеют это поле.

POST /auth/logout: должен инвалидировать refreshToken cookie на стороне сервера и вернуть Set-Cookie с expired cookie.

L. Potential Breaking Changes
Изменение backend	Что сломается
Переименование isAdmin → role: "admin"	Всё ролевое роутинг: auth.js, tickets.js, admin/dashboard.js
